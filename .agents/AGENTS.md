# 宇宙クイズ-AI プロジェクト開発ルール

このファイルは、本プロジェクト「宇宙クイズ-AI」でエージェントが開発を行う際の基本コンテキストとルールを定義しています。

## 1. アプリケーション概要とターゲット
- **アプリ名**: 宇宙クイズ-AI
- **ターゲット層**: 宇宙が大好きな未就学児（4〜6歳）、小学校低学年（7〜9歳）
- **想定デバイス**: タブレットでのタッチ操作に最適化したレスポンシブWebアプリ

## 2. クイズモードと設計制約
- **AI自動生成モード (AIのひみつクイズ)** と **手作りモード (おとうさん・おかあさんのクイズ)** が存在。
- **難易度**:
  - `easy` (やさしい): 漢字は一切使用禁止（すべて「ひらがな・カタカナ」）。
  - `medium` (ふつう): 簡単な漢字のみ許可、ただしすべての漢字に `<ruby>` タグによるルビ（ふりがな）必須。
  - `hard` (むずかしい): 漢字使用可、ただしすべての漢字に `<ruby>` タグによるルビ必須。
- **データ保存**: ユーザー情報や回答履歴、獲得バッジ、手作りクイズはすべてクライアントの `LocalStorage` に保存。

## 3. 音声・BGMシステム（重要）
- **実装ファイル**: `src/utils/audio.js`
- **設計方針**: 外部のMP3やWAVファイルは使用せず、**Web Audio API を用いてプログラム内で波形を合成する（シンセサイズ）** ことでBGMや効果音（SE）を鳴らします。
- 既存音源:
  - 効果音: クリック音、正解音、不正解音
  - BGM: 宇宙アンビエントBGM、8bitゲーム風BGM（ピコピコ音楽）
- 自動再生制限（Autoplay Policy）を考慮し、ユーザーの初回のインタラクション（クリック/タップ）で `AudioContext` を初期化・再開します。
  - **重要（UX設計）**: ブラウザ制限を自然にクリアするため、アプリの起動直後には「ウェルカム画面（スタート画面）」を表示し、ユーザーが必ず「スタートボタン」などのインタラクション操作を行ってからタイトル/メイン画面へ遷移する動線にします。この最初のタップ操作のハンドラー内で `audio.init()` および `audio.startBgm()` を呼び出し、確実にBGMの再生が始まるようにします。

## 4. インフラ & デプロイ構成
- **フロントエンド**: AWS Amplify Hosting
  - 環境変数 `VITE_API_GATEWAY_URL` に Lambda のエンドポイントURLを設定してビルド（出力先: `dist`）。
- **バックエンド**: AWS Lambda (Node.js 18+、ライブラリ依存ゼロ)
  - `lambda/index.js` が Gemini API と直接HTTPS fetchで連携し、バリデーションとリトライを実施。
  - 環境変数 `GEMINI_API_KEY` を Lambda 側に設定。
- **APIエンドポイント**: API Gateway または Lambda 関数 URL。

## 5. アセット・アイコン管理とキャッシュ対策
- **オリジナルアイコンの設定**:
  - PCブラウザやモバイル端末でブックマーク/ホーム画面に追加された際の視認性向上のため、オリジナルの `favicon.png` と `apple-touch-icon.png` を `public/` フォルダに配置して必ず設定します。
- **キャッシュ対策（Cache Busting）**:
  - 既存ユーザーのブラウザキャッシュによって古いアイコンが表示され続けるのを防ぐため、`index.html` 内のアイコンリンクの末尾には `?v=1` のようなバージョンクエリパラメータを必ず付与します。

## 6. 後方互換性とデータマイグレーション（LocalStorage）
- **データ構造の更新時**:
  - LocalStorage に保存するデータ構造を更新する（例: 配列からオブジェクトの配列へ変更するなど）場合、必ず既存ユーザーのデータを新構造に自動変換するマイグレーション処理を読み込み時（Get処理等）に記述します。
  - IDの変更や体系の変更が伴う場合は、古いIDから新しいIDへの変換マップ（`idMap`）を用意し、過去に獲得したデータが引き継がれるように保証してください。

## 7. レイアウト設計とスクロール・フィット制御
- **画面サイズのスクロール保証と二重スクロール競合の回避**:
  - モバイルや縦幅が著しく狭い環境、またはコンテンツがコンテナサイズをオーバーした際に、コンテンツが画面外に隠れて操作不能にならないよう、全体のスクロールを許可する設計を原則とします。
  - **重要（二重スクロールロックの防止）**: PC/タブレット通常表示時など、全体サイズが一定幅（max-height: 768px）に収まる環境では、親コンテナ（`.app-container`）の `overflow-y` は `hidden` とし、中の個別画面（`.scrollable-content` や `.test-container`）にスクロールを完全に委ねます。両方に `overflow-y: auto` が存在すると、外側がスクロールを奪って内側がスクロールできなくなる（スクロールフリーズ）ため絶対に避けてください。
  - モバイル環境（高さを `auto` にするメディアクエリ等）でのみ、親コンテナを `overflow: visible` にして `body` 全体でスクロールさせる設計とします。
  - 各スクリーンコンポーネントには `min-height: 100%` や `flex-shrink: 0` を指定し、狭い空間内で要素が不自然に縮んで潰れるのを防ぎます。
- **引っ張りスクロール防止と overscroll-behavior**:
  - ブラウザ全体の引っ張りバウンスを防ぐには、`html, body` に `overscroll-behavior-y: none` を適用してください。
  - **注意**: 親コンテナ（`.app-container`）や途中の親要素に `overscroll-behavior-y: none` を指定すると、ブラウザによっては子要素のスクロールイベントがフリーズしてしまう原因になります。個別画面のスクロール領域で親へのバウンス伝播を防ぐ場合は、スクロール可能な子要素自体に `overscroll-behavior-y: contain` を指定し、親コンテナでの過度な `none` 指定は避けてください。
- **タブレット等でのフィット感とリバランス**:
  - iPad等の特定の高さ制限（最大高さ 768px）の中で、スクロールさせずに1画面に綺麗に収めたい場合は、不用意に `overflow: hidden` で切り取るのではなく、要素間のギャップ（`gap`）、コンテナの `padding`、文字サイズを段階的に縮小・調整してバランスを取ります。
  - ただし、余白を削りすぎると高級感が失われるため、全体のデザインバランスを損なわない範囲でのリバランスを行います。
- **アセット完成時・獲得時の子供向けビジュアルUX**:
  - 探査機やロケット等の組み立てプロジェクトが完成した際（本物の写真がある場合）は、間接的な切り替えUIを挟まず、最初からリアルな宇宙写真をインパクト重視で表示します。
  - 写真をタップすると画面全体（フルスクリーンモーダル）に拡大（ズーム）する機能を実装します。
  - 拡大モーダルを閉じる際、閉じるボタンがはみ出したり見切れたりしても操作できるよう、**「画面のどこを（写真や背景を）タップしても元の画面に戻れる」** 設計を義務付けます。
  - **重要（完成後のパーツ見返し保証）**: プロジェクトが完成して完成体がメイン表示となった後も、ユーザーがそれまでに集めた個別パーツの解説や絵文字が見えなくなって寂しい思いをしないよう、詳細モーダル等の内部で各パーツをタブやミニスロットで切り替えて、いつでも説明文やイラストを読み返せる「図鑑モード（見返しスロット）」を必ず実装します。

## 8. Strict Mode / 二重マウントに対する副作用の防御設計
- **二重実行の防止**:
  - 結果画面のマウント時など、`useEffect` 内でデータの追加（LocalStorageの保存等）、購入処理、バッジ付与などの破壊的な副作用を実行する場合、React 18の Strict Mode（開発環境）による二重マウントを想定しなければなりません。
  - `useRef` を用いた「実行済みロックフラグ（例: `const earnedRef = useRef(false);`）」を定義し、副作用が**絶対に1回だけ**しか実行されないように確実に防衛してください。

## 9. フロント・バックエンド（Lambda）のクイズ同期設計
- **AIパラメータと同期**:
  - フロントエンドでAIクイズ生成のパラメータ（3択から4択への変更、検定モードフラグ等）を追加・変更する場合、必ずバックエンド側（`lambda/index.js`）にも同様のパース・プロンプト生成・バリデーションロジックを追加し、両者が完全に同期して動作するようにします。
  - 開発時の混乱を防ぐため、Vite環境変数がない場合はローカルのフォールバック用固定クイズが動作することを保証し、ローカル確認の手順（`.env.local` へのキー設定など）を明確にしておきます。

## 10. 外部API（LLM等）のローカル検証とシミュレーション
- **実キーを用いたローカル事前テスト**:
  - 外部APIとの通信処理やモデルの指定を変更する際、推測による本番デプロイの繰り返しは厳禁とします。必ずローカルにテストスクリプトを作成し、実際のAPIキーを用いた「200 OK」の動作確認とエラー時のレスポンス詳細を取得して検証してください。
- **インフラ制約のシミュレーション**:
  - Lambda等のタイムアウト時間（例: 3秒制限など）がある場合は、ローカルテストでも `Promise.race` やタイマー等を用いて本番の制約を再現し、制限時間内に確実に処理が完了するか実行速度を実測・確認してください。
- **Raw HTTP Fetch でのパラメータ記法 (Casing)**:
  - JavaScript SDKを使わず直接 HTTP raw fetch を行う場合、APIが期待するフィールドケース（通常は `generation_config` や `response_mime_type` などのスネークケース）を厳密に確認し、Casingミスによる 400 エラーを防止してください。

## 11. 生成AIクイズの重複・テーマ類似排除設計
- **具体的なコンテンツ履歴の引き渡し**:
  - AIクイズ生成において、過去の出題ID（ハッシュ値など）だけを渡してもAIはテーマを理解できません。連続して同じような趣旨の問題（例：太陽の動きが5連続など）が出るのを防ぐため、必ず「出題した問題の具体的なテキスト（またはキーワード）」を履歴（`excludeQuestions`）としてAPIに送信する設計にしてください。
  - プロンプト内には「最近出題された問題リストとテーマ・趣旨が重複しないこと」を記述し、AIに明確に異なるジャンルの出題を義務付けてください。

## 12. ルビ（ふりがな）付き文字列のプレーンテキスト化ルール
- **ふりがなの完全除去**:
  - 問題文や解説文からHTMLルビタグを除去してプレーンテキスト（ハッシュ値作成や重複比較用の文字列）にする場合、単にHTMLタグをストリップするだけでは `<rt>` タグ内のふりがな文字列が親文字と合体して残ってしまいます。
  - 必ず最初に `replace(/<rt>.*?<\/rt>/g, '')` を用いてふりがなの内容ごと消去した後に、残りのHTMLタグ `replace(/<[^>]*>/g, '')` を除去する2段構えの処理を義務付けます。

## 13. 音声読み上げ（Speech Synthesis）の精度・イントネーション向上ルール
- **助詞と句読点のハック**:
  - ひらがな表記の助詞「は」「へ」の誤読を防止するため、文末・スペース・読点の直前にある「は」「へ」を一時的に「わ」「え」に置換して発話テキストを生成します。
  - 音声合成において確実に「一呼吸（ポーズ）」を置かせるため、読点「、」および単語間のスペースは句点「。」に変換・統合してから音声エンジンへ渡します（音声エンジンは「。」で確実にポーズをとるためです）。
- **特異ワードのアクセント補正**:
  - 音声エンジンの仕様上、平坦なアクセントで読まれやすい単語（「日本」「ベガ」「名前」「時間」「お星様」など）は、辞書引きアクセント（頭高・中高など）を発生させるために、読み上げ専用テキスト内で一時的に別の表記（「にっぽん」「ヴェガ」「名前」「時間」「お星さま」など）へ置換するマッピング処理を行います。
- **ルビタグ内の送り仮名消失の防止**:
  - `<ruby>浮<rt>う</rt>かぶ</ruby>` のように送り仮名がルビタグに含まれる場合、HTMLテキストからルビ部分をパースして平滑化する際に、送り仮名にあたるテキストノード（`rt` タグより後ろの文字列）が消失しないよう、ルビ要素の子ノードを詳細走査して送り仮名を保護・結合するロジックを適用します。
- **アクセント調整の限界と過剰なハックの抑止**:
  - 音声合成（SpeechSynthesis）の発音高低（アクセント）は、実行中のブラウザやOSの音声エンジンに強く依存します。スペースの挿入や、イントネーション調整を目的とした意図的ではない同音異義語への漢字置換（「いち」を「位置」にするなど）、過度な長音化（「いーち」など）は、一部の環境で動作しなかったり不自然な発音を招く原因になります。
  - したがって、発音アクセントの調整は標準的な辞書引きが働く「標準の漢字・平仮名・カタカナ表記」の範囲に留め、極端なテキスト表記ハックの過剰な適用は避けてください。

## 14. 画面遷移・モーダル閉動作時の誤タップ（チャタリング・突き抜け）防止設計
- **遷移およびモーダル閉動作直後の入力ロック**:
  - クイズ結果の決定や「つぎへ すすむ」ボタンが押された遷移時、およびズームモーダル等のポップアップを閉じる操作（「とじる」ボタン等のタップ）が行われた直後に、同じ座標にある次のインタラクティブ要素（回答選択肢など）が誤ってタップされる現象（チャタリングやクリック突き抜け）を防止します。
  - 画面遷移やモーダルクローズ処理が開始された際に、一時的なロック状態（`isTransitioning` など）を `true` にし、最初の `500ms〜800ms`（推奨 `600ms`）の間はクリックやタッチイベントを無視するガードロジックを実装します。
  - タイマーを管理する際は、メモリリーク防止のため、コンポーネントのアンマウント時に必ず `clearTimeout` によるクリーンアップを行ってください。

## 15. マーカー（Canvasペイント）の視認性設計
- **ハイライターとしての不透明度**:
  - 問題文やテキストの上になぞり書きするマーカー（Canvas描画）を実装する際、下に書かれているテキストが完全に隠れてしまわないよう、描画線のアルファ透過度（不透明度）は低め（`0.2` 付近）に設計してください。

## 16. 条件付きレンダリングと DOM/Canvas 初期化の防衛設計
- **ロード状態などを伴う ref 初期化の注意**:
  - ロード中画面（`loading === true`）など、Canvas や特定の DOM 要素が条件付きでレンダリングから除外されている場合、コンポーネントマウント直後の `useEffect` では `ref.current` が `null` になるため初期化に失敗します（1問目だけペンが機能しない等のバグの原因になります）。
  - このような要素を初期化する `useEffect` では、必ず `loading` 等の条件フラグを依存配列（Dependency Array）に含め、かつ `!loading` 条件下で初期化関数（`initCanvas` など）を走らせるガードロジックを実装してください。

## 17. 非同期・バックグラウンドロード時のUI・音声リセット防止設計
- **エフェクト依存配列の防衛**:
  - バックグラウンドで非同期的にデータがロードされ、配列（例: `quizzes`）に要素が順次追加されていく場合、配列全体の参照変更を検知して不要な副作用が再発火するのを防止する必要があります。
  - 音声読み上げ（`SpeechSynthesis`）などの自動トリガーエフェクトの依存配列（Dependency Array）には、配列全体（`quizzes`）を指定せず、**現在アクティブに表示している特定のオブジェクト（`quizzes[currentIdx]`）を指定**し、裏でのデータ追加による音声や画面表示のリセットを完全に防いでください。
- **総問題数・進捗率の固定表示**:
  - 非同期にロードされる最中、ロードされた分だけ総問題数（`quizzes.length`）が増えていくと、進捗率（プログレスバー）の巻き戻りやカウンター表示の変動（例: 「ぜんぶで 2 もん」から「5 もん」へカクカク増えるなど）が発生してUXを損ねます。
  - AI自動生成モードなど最終的な問題数が決定している場合は、ロード途中であっても定数（例: `totalQuestions = 5`）を用いて総数表示と進捗率をマスク（固定表示）してください。
- **取得未完了（遅延）への境界ガード**:
  - ユーザーの回答スピードが非同期ロード速度を上回り、まだロードされていない問題に進もうとした場合に備え、`!currentQuiz` の条件で一時的なロード画面（「つぎのもんだいを作っているよ」等）を表示し、データの追加検知に伴い自動で問題画面へシームレスに切り替わるガードロジックを実装してください。
